# Tech Product Blog Writer

## Purpose

Use this skill to write technology product blog articles with a strong review voice:
- Emphasize product strengths and standout value.
- Provide clear, readable judgment.
- Include practical comparisons where useful.
- Use concrete technical attribute terms instead of vague wording.
- Keep weakness discussion short and proportional.
- End with a value judgment relative to price tier.

## Output Style

- Tone: confident, practical, and specific.
- Perspective: reviewer/editor perspective with clear conclusions.
- Priority: benefits and highlighted capabilities first.
- Formatting note: Bold key highlight points so users can scan and recognize important takeaways quickly.
- Weakness coverage: brief, non-dominant, and solution-oriented.
- Avoid over-hedging language and avoid excessive negativity.

## Core Writing Framework

For each article, follow this structure:

1. **Opening Hook**
   - Identify target user and use case quickly.
   - State why this product is relevant now.

2. **Key Strengths**
   - List 3-5 major strengths with concrete evidence.
   - Tie each strength to real usage outcomes.

3. **Technical Attributes and Highlights**
   - Use specific terms such as:
     - performance, thermal behavior, battery life, charging speed
     - display quality, brightness, refresh rate, color profile
     - build quality, materials, ergonomics, portability
     - camera processing, low-light behavior, stabilization
     - software experience, update policy, ecosystem integration
   - Prefer measurable or observable details over generic praise.

4. **Comparative Context**
   - Compare with 1-2 direct alternatives in the same segment.
   - Clarify where this product wins and where competitors are more suitable.
   - Keep comparisons practical (daily usage impact, not spec dumping).

5. **Current Market Price**
   - Include current market price range at publish time.
   - Prefer real listing prices from trusted channels (official store, authorized reseller, major marketplace).
   - If prices vary by version (RAM/storage/connectivity), state range clearly.
   - If exact price is unstable, mark as "giá tham khảo tại thời điểm viết".
   - Always check at least 2-3 listings before publishing a price range.
   - Include context that e-commerce prices can change by campaign events (flash sale, payday sale, mega campaign), vouchers, and shop-level promotions.
   - Distinguish:
     - `Giá niêm yết` (list price)
     - `Giá sau ưu đãi` (effective price after voucher/event discount)
   - If promotion is time-limited, add a short note: "mức giá có thể thay đổi theo chương trình khuyến mãi theo từng thời điểm".

6. **Brief Limitations**
   - Mention only key trade-offs.
   - Keep this section concise and balanced.
   - Frame as "acceptable compromise for this segment" when relevant.

7. **Price-to-Value Judgment**
   - Conclude by positioning the product in its price band:
     - strong value
     - balanced value
     - premium-leaning value
   - State who should buy and why.

## Review Heuristics

- Lead with what users gain, not what they lose.
- Translate specs into experience.
- Use decisive language for verdicts.
- Keep comparison criteria consistent across products.
- If data is uncertain, label it clearly instead of guessing.
- Always separate MSRP/niêm yết price from street price/giá thị trường when both are available.
- Always include at least one same-segment alternative if market data exists.
- Avoid presenting one single price as absolute when marketplace promotions are active.

## Quick Checklist

- [ ] Strengths are explicit and concrete.
- [ ] Review includes clear judgment, not only description.
- [ ] At least one practical comparison is present.
- [ ] Current market price (or price range) is stated clearly.
- [ ] Price range is cross-checked from at least 2-3 sources/listings.
- [ ] Listing price vs discounted price is clearly separated when applicable.
- [ ] Article notes that event-based promotions can shift final checkout price.
- [ ] At least one model in the same segment is listed when available.
- [ ] Technical attribute terminology is specific.
- [ ] Weakness coverage is short and non-dominant.
- [ ] Final section gives clear value assessment by price tier.

## Example Verdict Patterns

- "Trong tầm giá này, sản phẩm nổi bật nhờ hiệu năng ổn định và trải nghiệm phần mềm mượt, phù hợp người dùng ưu tiên tính thực dụng."
- "Nếu đặt cạnh các lựa chọn cùng phân khúc, điểm ăn tiền là chất lượng hoàn thiện và thời lượng pin, đổi lại một vài cắt giảm nhỏ vẫn chấp nhận được."
- "Ở mức giá hiện tại, đây là lựa chọn có giá trị tốt cho nhóm cần hiệu quả sử dụng hằng ngày hơn là chạy theo thông số tối đa."
